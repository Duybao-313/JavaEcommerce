## 🔐 Token Configuration

**File:** `application.yaml` (Lines 16-19)

```yaml
jwt:
  secret: "q9v8G3s2r4Yt1KfVb6s9X2mQp7L0uH8nZ5yR0wVxC1aB7dE4fG6hJkL9mN0oPqR2sT3uVwXyZ+AbCdEfGhIjKlMBaoBaoBao"
  token-duration: 6000              # 100 minutes (1.67 hours)
  refresh-duration: 60000           # 16.67 hours (~1 day)
```

### Token Lifetime Strategy:
- **Access Token (JWT):** Valid for **6000 seconds (~100 minutes)**
- **Refresh Token Window:** Valid for **60000 seconds (~16.67 hours)**
  - Allows token refresh as long as it's within the refresh duration
  - Even if access token is expired, can still refresh if within this window
  - After 16.67 hours, must login again

## 🔄 Token Refresh Flow (When Token Expires)

### Endpoint: `POST /auth/refresh`
**File:** `AuthenticationController.java` (Lines 75-82)

✅ **PUBLIC endpoint** - No authentication required  
⚠️ **BUT** - Token itself must be valid (within refresh duration)

#### Request Format:
```json
{
  "token": "eyJhbGc..."  // Expired or about-to-expire JWT token
}
```

#### Processing Steps (JwtService.refreshToken):

**1. Token Validation** (Line 87)
```
Call VerifyToken(token, isRefresh=true)
  ├─ Verify JWT signature with HMAC512
  ├─ Check if token was issued ≤ 60000 seconds ago (refresh window)
  ├─ Check if token is in InvalidatedToken table (blacklist)
  └─ If ANY check fails → Throw AppException(TOKEN_INVALID)
```

**2. Token Invalidation** (Lines 88-92)
```java
var jit = signedJWT.getJWTClaimsSet().getJWTID();
Date expiration = signedJWT.getJWTClaimsSet().getExpirationTime();
InvalidatedToken invalidatedToken = 
  InvalidatedToken.builder()
    .id(jit)                    // Store JWT ID
    .expiryTime(expiration)     // Store original expiration
    .build();
invalidatedTokenRepository.save(invalidatedToken);  // Add to blacklist
```

**3. Generate New Token** (Lines 93-96)
```java
var username = signedJWT.getJWTClaimsSet().getSubject();
User user = userRepository.findByUsername(username)
  .orElseThrow(() -> new AppException(ErrorCode.UNAUTHENTICATED));
var refreshToken = generateToken(user).getToken();  // New 100-min token
```

#### Response Format:
```json
{
  "success": true,
  "code": 200,
  "message": "tao thanh cong",
  "data": {
    "token": "eyJhbGc...",         // NEW access token (valid 100 min)
    "expiryDate": "2026-04-24T16:45:00.000Z"
  },
  "timestamp": "24-04-2026 15:04:09"
}
```

## 🛡️ Token Validation Flow (Request with Token)

### Endpoint: Any endpoint requiring authentication
**File:** `SecurityConfigV2.java` (Lines 33-42)

```
Incoming Request
  ↓
Extract "Authorization: Bearer {token}" header
  ↓
JwtDecoderConfig.decode() (Lines 35-44)
  ├─ Call jwtService.introspect(token)
  │   ├─ Verify signature
  │   ├─ Check expiration time  (NOT refresh window, actual expiration)
  │   ├─ Check InvalidatedToken table
  │   └─ Return { valid: true/false }
  │
  ├─ If NOT valid → Throw JwtException
  └─ If valid → Parse and return JWT
  ↓
jwtConverter.convert(jwt) → Extract claims, load User from DB
  ↓
Set SecurityContext with authenticated User
  ↓
Allow request to proceed
```

### Error Cases:
- ❌ **Invalid signature** → 401 Unauthorized
- ❌ **Token expired** → 401 Unauthorized  
- ❌ **Token blacklisted** (in InvalidatedToken table) → 401 Unauthorized
- ❌ **Missing Authorization header** → 401 Unauthorized

## 🔒 Token Validation Differences

### For `VerifyToken(token, isRefresh=false)` - Request Validation
```java
Date expiration = signedJWT.getJWTClaimsSet().getExpirationTime();
// Uses actual expiration time from token
// If current time > expiration time → INVALID
```

### For `VerifyToken(token, isRefresh=true)` - Refresh Validation
```java
Date expiration = new Date(
  signedJWT.getJWTClaimsSet()
    .getIssueTime()
    .toInstant()
    .plus(refreshDuration, ChronoUnit.SECONDS)  // 60000 seconds
    .toEpochMilli()
);
// Uses: issueTime + refreshDuration
// If current time > issueTime + 60000 → INVALID
// Allows refresh even if token is already expired
```

**Example:**
```
Token issued: 2026-04-24 10:00:00
Token exp: 2026-04-24 11:40:00 (100 min duration)
Refresh window: 2026-04-25 02:40:00 (issueTime + 16.67 hours)

Scenario A: Request at 2026-04-24 11:30:00
  ├─ Token valid? YES (within 100 min)
  ├─ Can use normally ✓
  └─ Can refresh? YES (within refresh window)

Scenario B: Request at 2026-04-24 12:00:00 (token expired)
  ├─ Token valid? NO (past expiration)
  ├─ Normal request fails with 401
  ├─ Can refresh? YES (within refresh window)
  └─ Frontend calls /auth/refresh → New token generated

Scenario C: Request at 2026-04-25 03:00:00 (refresh window closed)
  ├─ Can refresh? NO (past refresh window)
  ├─ /auth/refresh returns 401
  └─ User must login again
```

## 👤 User Principal Resolution

**File:** `jwtConverter.java` (Lines 26-54)

When a token is validated, Spring Security needs to understand "WHO" is making the request:

```java
public AbstractAuthenticationToken convert(Jwt jwt) {
  Object useridObj = jwt.getClaim("userid");
  
  if (useridObj != null) {
    try {
      Long userId = Long.parseLong(useridObj.toString());
      Optional<User> opt = userRepository.findById(userId);
      if (opt.isPresent()) {
        User user = opt.get();  // Load full User from DB
        return new UsernamePasswordAuthenticationToken(user, jwt, authorities);
        // ✓ Principal is User object (full details available)
      }
    } catch (Exception ex) {}
  }
  
  // Fallback: Principal is just username string
  String principalName = jwt.getSubject();
  return new UsernamePasswordAuthenticationToken(principalName, jwt, authorities);
}
```

### Result:
- **Ideal:** `@AuthenticationPrincipal User user` → Returns full User object with DB data
- **Fallback:** If User not found → Returns string username

## 📊 Token Refresh Timeline Example

```
10:00:00 - User logs in
  ├─ Access token issued (valid until 11:40:00)
  ├─ Refresh window until 02:40:00 next day
  └─ Stored in localStorage

11:30:00 - User makes request
  ├─ Token is valid (within 100 min)
  ├─ Request succeeds
  └─ No refresh needed

11:50:00 - User makes another request
  ├─ Token is valid (still 50 min remaining)
  ├─ Request succeeds
  └─ No refresh needed

12:05:00 - User makes request (token expired!)
  ├─ Token validation fails (past 11:40:00)
  ├─ Backend returns 401 Unauthorized
  ├─ Frontend catches 401
  ├─ Calls POST /auth/refresh with old token
  ├─ Refresh validation succeeds (within 16.67 hour window)
  ├─ New token issued (valid until 13:45:00)
  ├─ Stored in localStorage
  └─ Original request retried with new token → SUCCESS

02:50:00 NEXT DAY - User makes request
  ├─ Token expired (past original time)
  ├─ Refresh window CLOSED (past issueTime + 16.67 hours)
  ├─ POST /auth/refresh returns 401
  ├─ No token available
  └─ User must login again
```

## ✅ Frontend Implementation Verification

### File: `authApi.js`

#### ✓ Refresh Function (Lines 117-136)
```javascript
export async function refreshAuthToken(token = getStoredRefreshToken()) {
  if (!token) {
    throw new Error('Không có token để làm mới')
  }

  const response = await fetch(`${API_BASE}/auth/refresh`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ token }),  // ✓ Correct format
  })

  const apiResponse = await parseResponse(response)
  const normalized = {
    ...apiResponse,
    data: normalizeAuthData(apiResponse),
  }

  persistAuthResult(normalized)  // ✓ Saves new token
  return normalized
}
```

**Status:** ✅ **CORRECT** - Sends token in correct format, saves new token

#### ✓ Auto-Refresh on 401 (Lines 138-167)
```javascript
export async function authFetch(input, init = {}) {
  const headers = new Headers(init.headers || {})
  const token = getStoredToken()

  if (token) {
    headers.set('Authorization', `Bearer ${token}`)  // ✓ Correct format
  }

  const response = await fetch(input, {
    ...init,
    headers,
  })

  if (response.status !== 401) {
    return response  // ✓ Only refresh on 401
  }

  const refreshed = await refreshAuthToken()
  const nextHeaders = new Headers(init.headers || {})
  const nextToken = refreshed?.data?.token || getStoredToken()

  if (nextToken) {
    nextHeaders.set('Authorization', `Bearer ${nextToken}`)
  }

  return fetch(input, {
    ...init,
    headers: nextHeaders,
  })
}
```

**Status:** ✅ **CORRECT** - Automatically refreshes on 401, retries with new token

#### ✓ Token Storage (Lines 105-115)
```javascript
export function persistAuthResult(payload) {
  if (!payload?.data) return

  const authData = payload.data
  localStorage.setItem('splitgo_auth', JSON.stringify(authData))

  if (authData.token) {
    localStorage.setItem('splitgo_token', authData.token)
    localStorage.setItem('splitgo_refresh_token', authData.refreshToken || authData.token)
  }
}
```

**Status:** ✅ **CORRECT** - Stores both token and refreshToken

## ⚠️ Important Notes

1. **Refresh Token is just the Old Token**
   - The `refreshToken` stored is actually the same as access token
   - Both call `/auth/refresh` with the same token value
   - Backend checks renewal window, not a separate refresh token

2. **Token Blacklist**
   - Old tokens are added to `InvalidatedToken` table after refresh
   - Prevents token reuse after refresh
   - Improves security

3. **Concurrent Requests During Refresh**
   - Current implementation may cause multiple refresh calls if several requests fail with 401 simultaneously
   - Not a critical issue for current use case
   - Future enhancement: Add token refresh mutex

4. **Login Response Structure**
   - Backend returns `jwt` field (JWT string), not `token`
   - Frontend's `normalizeAuthData()` handles mapping: `data.token = data.jwt`

5. **No Proactive Refresh**
   - Frontend only refreshes when 401 is received
   - Could improve UX by proactively refreshing before expiration
   - Current approach is simpler and works fine
